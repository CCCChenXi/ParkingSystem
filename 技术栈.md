# ParkingSystem 技术栈与方法总结

## 一、项目概述

智能停车管理系统，采用**客户端 + 管理端双端架构**，覆盖用户注册登录、附近停车场搜索、车位状态实时查询、停车订单生命周期、优惠券秒杀、钱包充值扣费等功能。管理端包含数据看板、车位/优惠券/用户管理。

## 四、缓存（三级架构）

```
请求 → Caffeine (L1) → Redis (L2) → MySQL (L3)
                    → 任一缓存失效自动降级下一层
```

### 4.1 Caffeine 本地缓存（7 个独立 Cache bean）

| Cache 名称 | 容量 | TTL | 用途 |
|-----------|------|-----|------|
| `couponAvailableCache` | 500 | 3 min | 可领取优惠券列表 |
| `parkingLotCache` | 1000 | 10 s | 停车场详情 |
| `vehicleCache` | 10000 | 2 min | 用户车辆列表 |
| `walletBalanceCache` | 10000 | 30 s | 钱包余额 |
| `couponDetailCache` | 500 | 5 min | 优惠券详情 |
| `couponStaticCache` | 500 | 5 min | 优惠券静态信息 |
| `parkingSpotCache` | 500 | 5 min | 车位列表 |

### 4.2 Redis 数据类型与用途

| 数据类型 | 用途 |
|---------|------|
| **String** | 会话缓存、短信验证码、登录失败计数器、账号锁定标记、JSON 对象缓存、秒杀库存计数器、月编辑次数、仪表盘计数 |
| **Hash** | 实时趋势数据（按月分 key）、管理端车位状态、可用车位数 |
| **Bitmap** | 车位占用状态（客户端，位运算实现 O(1) 查询） |
| **GEO** | 附近停车场搜索（`GEORADIUS` 按距离排序、限制 10 条） |
| **Set** | 秒杀抢购用户去重，防止重复领取 |

### 4.3 Redis 高级特性

| 特性 | 实现 | 用途 |
|------|------|------|
| 分布式锁 | Redisson `RLock` + Lua 脚本 | 车位状态重建防并发 |
| Lua 原子脚本 | `seckillCoupon.lua` / `ReleaseLock.lua` / `rebuildLot.lua` | 秒杀扣库存+去重原子操作、分布式锁释放 |
| 连接池 | Lettuce Pool（max-active=8） | Redis 连接复用 |
| 缓存穿透防护 | 空值缓存（停车场信息不存在时缓存空串） | 避免无效 key 穿透到 DB |
| 优雅降级 | `try-catch` ，非关键路径自动降级 | Redis 故障时不影响核心业务流程 |
| 临时表重建 | tempKey → `rename` 模式 | 避免缓存重建期间读到不完整数据 |
| 缓存失效广播 | RabbitMQ Fanout Exchange | 多实例部署时同步清除本地 Caffeine 缓存 |

## 五、消息中间件

### 5.1 RabbitMQ 架构

| Exchange | Type | Routing Key / Queue | 用途 |
|----------|------|--------------------|------|
| `seckill.order.exchange` | Direct | `seckill.order.queue` | 秒杀优惠券异步订单创建 |
| `cache.invalidate.exchange` | Fanout | 各实例独立 AnonymousQueue | 多实例缓存失效广播 |
| `parking.exchange` | Topic | `order.reserved` / `order.enter` / `order.settle` | 停车订单生命周期事件 |
| (管理端) | — | `parking.lot.cache.init` / `parking.spot.cache.retry` | 管理端缓存初始化与重试 |

### 5.2 消息流

**秒杀订单：**
```
Controller → Lua 原子扣减 → SeckillMessageProducer
    → Direct Exchange (seckill.order.exchange)
    → Queue (seckill.order.queue)
    → SeckillMessageConsumer → 异步写入 UserCoupon
```

**缓存失效广播：**
```
CouponService → CacheInvalidateProducer
    → Fanout Exchange (cache.invalidate.exchange)
    → 所有实例的 AnonymousQueue
    → CacheInvalidateConsumer → 清除本地 Caffeine 缓存
```

**停车订单生命周期：**
```
OrderEvent → Topic Exchange (parking.exchange)
    → order.reserved / order.enter / order.settle
    → OrderMessageConsumer → 异步处理
```

### 5.3 回滚与重试

| 场景 | 方案 |
|------|------|
| 超时未入场 | `ReservedOrderTTLQueue` + Dead Letter → `SpotReleaseRetryConsumer` 释放车位 |
| 缓存初始化失败 | `ParkingSpotCacheRetryEvent` 延迟重试队列 |
| 消费异常 | 日志记录，不阻塞主流程 |

## 九、设计模式与方法

### 9.1 架构模式

| 模式 | 实现 |
|------|------|
| **三层架构** | Controller → Service → Mapper |
| **门面模式** | `ParkingDataProvider` / `CouponDataProvider` 封装多级缓存读写逻辑 |
| **双端分离** | `client/` 客户端模块 + `admin/` 管理端模块，共享 `common/` 公共层 |

### 9.2 缓存策略

| 策略 | 实现 |
|------|------|
| **多级缓存** | Caffeine (L1) → Redis (L2) → DB (L3)，逐级降级 |
| **缓存穿透防护** | 空值缓存，DB 查询不到结果也缓存空串 |
| **缓存失效广播** | RabbitMQ Fanout 通知所有实例清除本地缓存 |
| **分布式锁重建** | Redisson `RLock` + tempKey → rename 防止重复重建 |
| **本地缓存预热** | `@PostConstruct` 在启动时加载优惠券库存到 Redis |
| **游标分页** | `WHERE (time, id) < (?, ?)` 二分法游标，避免深度分页 |

### 9.3 并发控制

| 场景 | 方案 |
|------|------|
| 秒杀扣库存 | Lua 脚本原子扣减（stock + 去重 Set 在同一条 Lua 中完成） |
| 普通优惠券领取 | 数据库 `UPDATE ... WHERE remain_stock > 0` 乐观锁 |
| 缓存重建 | Redisson `RLock` 互斥锁，tempKey → rename 原子切换 |
| 分布式锁释放 | Lua 脚本对比线程 ID 后删除，防止误删 |

### 9.4 容错与降级

| 场景 | 策略 |
|------|------|
| Redis 故障（关键路径：登录/注册） | 硬失败，抛 `BusinessException(500)` |
| Redis 故障（非关键路径：缓存查询） | 返回 null，自动降级到 DB |
| Redis 故障（秒杀） | 不允许直接成功，优先保证系统安全 |
| 管理端 Redis 故障 | 缓存仅作为加速，降级直接查 DB |
| GEO 搜索 Redis 故障 | 降级为 Haversine 距离公式 + 经纬度范围查询 |
| `@RabbitListener` 异常 | 日志记录，不阻塞主流程 |
| 全局异常兜底 | `@RestControllerAdvice` 捕获所有未处理异常，返回统一错误响应 |

### 9.5 事务与缓存一致性

| 手段 | 实现 |
|------|------|
| 事务后清理缓存 | `TransactionSynchronizationManager.registerSynchronization` + `afterCommit()` |
| Spring Cache | `@Cacheable` / `@CacheEvict` 声明式缓存管理 |
| 显式 Redis 删除 | 事务提交后手动 `stringRedisTemplate.delete(key)` |

### 9.6 其他设计

| 模式/方法 | 实现 |
|-----------|------|
| 全局异常处理 | `@RestControllerAdvice` 统一处理：`BusinessException` → 指定 code、校验异常 → 400、系统异常 → 500 |
| 统一返回格式 | `Result<T>` 包含 code / message / data，`PageResult<T>` 包含分页信息 |
| 配置分离 | `@ConfigurationProperties(prefix="jwt")` 注入 JWT 配置 |
| 魔法值管理 | 所有硬编码字符串/数字提取到 `*Constant.java` |
| 自动填充 | `MetaObjectHandler` 统一处理实体创建/更新时间 |
| GEO 附近搜索 | Redis `GEORADIUS` 按距离排序，失败降级为 Haversine 公式计算 |

## 十、代码亮点

### 10.1 CacheResult 统一缓存结果 + 优雅降级

```java
public enum CacheResult<T> {
    HIT(T),     // 缓存命中
    MISS(null), // 缓存未命中（穿透时）
    ERROR(null) // 缓存异常（降级到下一层）
}
```

`CacheResult` 作为所有缓存操作的统一返回类型，让调用方根据结果类型决定走 DB 还是抛异常：

- **关键路径（登录/注册）**：缓存 ERROR 时硬失败抛 `BusinessException(500)`，保证系统安全
- **非关键路径（缓存查询）**：缓存 ERROR 时自动降级到 DB，业务不受影响
- **秒杀场景**：Redis 不可用时不放行，优先系统安全

### 10.2 三级缓存门面模式

`ParkingDataProvider` 和 `CouponDataProvider` 封装了从 Caffeine → Redis → DB 的完整读写链路，包含以下机制：

- **缓存穿透防护**：DB 查询空结果也缓存（NULL_MARKER 对象或空串），防止恶意 key 击穿
- **缓存雪崩防护**：tempKey → rename 原子重建，避免并发重建期间读到不完整数据
- **分布式锁互斥**：Redisson `RLock` 保证同时只有一个实例重建缓存
- **双检锁**：获取锁后二次检查缓存是否已被其他实例重建
- **本地预热**：`@PostConstruct` 启动时加载热点数据
- **多实例失效广播**：更新 DB 后发 RabbitMQ Fanout 消息，所有实例清除本地 Caffeine

### 10.3 Lua 脚本原子操作

**`seckillCoupon.lua`** — 三个操作在一个脚本中原子完成：
```lua
if sismember(KEYS[2], ARGV[1]) == 1 then return 2 end  -- 去重检查
local stock = tonumber(redis.call('get', KEYS[1]))
if stock >= 1 then
    decr(KEYS[1]); sadd(KEYS[2], ARGV[1]); return 1     -- 扣库存 + 记录用户
end
return 0                                                  -- 库存不足
```

**`ReleaseLock.lua`** — 对比线程 ID 后删除，防止误删其他线程锁。

### 10.4 Bitmap 车位状态 — O(1) 位运算

利用 Redis Bitmap 存储车位占用状态，通过字节数组位运算实现高效读写：

```java
byte[] bitmap = stringRedisTemplate.execute((RedisCallback<byte[]>) conn ->
    conn.stringCommands().get(key.getBytes()));
for (int i = 0; i < bitmap.length * 8; i++) {
    boolean occupied = (bitmap[i / 8] & (1 << (7 - i % 8))) != 0;
}
```

相比传统 String/Set 存储，Bitmap 节省数十倍内存，且 SETBIT/GETBIT 时间复杂度 O(1)。

### 10.5 事务后置一致性

使用 `TransactionSynchronizationManager.registerSynchronization` + `afterCommit()` 确保非 DB 操作在事务提交后才执行：

```java
TransactionSynchronizationManager.registerSynchronization(
    new TransactionSynchronization() {
        public void afterCommit() {
            // Redis 状态变更、WebSocket 通知等在事务提交后执行
            stringRedisTemplate.opsForValue().set(spotKey, "0");
            notificationPublisher.publish(userId, messageVO);
        }
    });
```

核心价值：**DB 回滚时 Redis 不会脏写**，保证最终一致性。

### 10.6 GEO 附近搜索 + Haversine 降级

```java
// 优先使用 Redis GEO
Set<String> nearby = stringRedisTemplate.opsForGeo()
    .radius(PARKING_GEO_KEY, circle, args);

// Redis 故障时降级为 Haversine 距离公式 + 经纬度范围查询
double distance = DistanceUtil.distance(lat1, lng1, lat2, lng2);
```

双重保障，GEO 搜索的可用性不依赖 Redis。

### 10.7 游标分页（避免深度分页）

```java
// 使用 (time, id) 二元组游标，不用 offset
List<Coupon> list = couponMapper.selectList(new LambdaQueryWrapper<Coupon>()
    .lt(Coupon::getCreateTime, lastTime)
    .or()
    .eq(Coupon::getCreateTime, lastTime)
    .lt(Coupon::getId, lastId)
    .orderByDesc(Coupon::getCreateTime, Coupon::getId)
    .last("LIMIT " + pageSize));
```

传统 `LIMIT 100000, 10` 的深度分页性能极差，游标分页始终扫描少量数据。

### 10.8 三层拦截器链

| 层级 | 拦截器 | 职责 |
|------|--------|------|
| 第一层（所有路径） | `RefreshTokenInterceptor` | 解析 JWT Token，刷新 Redis 会话有效期；不阻断请求 |
| 第二层（客户端 API） | `JwtTokenUserInterceptor` | 校验 Token 有效性，检查 Redis 会话，设置 `UserHolder` |
| 第二层（管理端 API） | `JwtTokenAdminInterceptor` | 同上，设置 `AdminHolder` |

设计优点：Token 刷新与权限校验分离，第一层拦截所有路径保证即使静态资源也能刷新会话，第二层按需校验身份。

### 10.9 RabbitMQ 重试 + 死信队列

超时未入场的订单通过死信队列自动释放车位：

```
OrderReserved → Delay Queue（30分钟 TTL）
    → Dead Letter Exchange
    → SpotReleaseRetryConsumer → 释放车位 + 通知用户
```

管理端缓存初始化失败通过延迟重试队列自动恢复，最大重试 3 次 + 告警。

### 10.10 双端分离 + 常量规范

```
common/   共享：实体、配置、常量、工具类、拦截器、MQ、缓存
client/   客户端：用户视角的停车场、订单、优惠券、钱包 API
admin/    管理端：管理视角的看板、车位、优惠券、用户管理 API
```

- 10 个 `*Constant.java` 类杜绝魔法值
- `AGENTS.md` 代码规范覆盖命名、异常、日志、校验、Redis 使用
- 所有 Redis Key 遵循 `{模块}:{功能}:{后缀}` 格式
- TTL 常量命名带单位（`_TTL_MIN` / `_TTL_DAY` / `_HOUR`）

## 十二、代码规范

| 规范 | 要求 |
|------|------|
| 异常处理 | 业务失败抛 `BusinessException(code, msg)`，不直接 `return Result.fail` |
| 日志级别 | 系统异常 `log.error`，业务异常 `log.warn`，流程日志 `log.info` |
| 入口日志 | 每个 Controller 方法必须写 `log.info("操作描述: {}", 关键参数)` |
| 参数校验 | 用 `@Pattern` / `@NotBlank`，Controller 加 `@Validated`，Service 不做校验 |
| 常量管理 | 魔法值提取到 `*Constant.java`，Redis Key 格式 `{模块}:{功能}:` |
| 类型一致 | VO/DTO 字段类型必须与 Entity 一致（`Long` 不写成 `Integer`） |
| Redis 使用 | 优先用 `RedisService`，不要直接用 `StringRedisTemplate`（验证码模块例外） |
| 命名规范 | 类/接口大驼峰，方法/参数小驼峰，常量全大写+下划线，包名全小写 |
