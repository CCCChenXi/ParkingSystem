## 三、Controller 层规范

```java
@Slf4j
@RestController
@RequiredArgsConstructor
@RequestMapping("/prefix")
public class XxxController {

    private final XxxService xxxService;

    @GetMapping("/{id}")
    public Result get(@PathVariable Long id) {
        log.info("查询XX: id={}", id);
        return xxxService.get(id);
    }

    @PostMapping
    public Result create(@Validated @RequestBody XxxDTO dto) {
        log.info("新增XX: name={}", dto.getName());
        return xxxService.create(dto);
    }

    @PutMapping("/{id}")
    public Result update(@PathVariable Long id,
                         @Validated @RequestBody XxxDTO dto) {
        log.info("更新XX: id={}", id);
        return xxxService.update(id, dto);
    }

    @DeleteMapping("/{id}")
    public Result delete(@PathVariable Long id) {
        log.info("删除XX: id={}", id);
        return xxxService.delete(id);
    }
}
```

**规则：**
- 所有方法返回 `Result`
- 每个方法第一行写 `log.info("操作描述: {}", 关键参数)`
- 注入 `private final` 字段，通过 `@RequiredArgsConstructor` 构造注入
- 写操作用 `@Validated @RequestBody DTO` 接收参数
- 路径变量用 `@PathVariable`，查询参数用 `@RequestParam`
- Service 层不做参数校验

## 四、Service 层规范

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class XxxServiceImpl implements XxxService {

    private final XxxMapper xxxMapper;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public Result xxxMethod(XxxDTO dto) {
        // 业务逻辑
        // 异常抛 BusinessException
    }
}
```

**规则：**
- 类注解顺序：`@Slf4j` → `@Service` → `@RequiredArgsConstructor`
- 只注入 `private final` 字段（构造注入）
- 写操作加 `@Transactional(rollbackFor = Exception.class)`
- 事务内修改 DB 后，缓存清理通过 `TransactionSynchronizationManager.afterCommit()` 延迟到提交后执行

```java
TransactionSynchronizationManager.registerSynchronization(
    new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            // 删除或重建缓存
        }
    });
```

## 五、Entity 层规范

```java
@Data
@TableName("snake_case_table")
public class XxxEntity {
    @TableId(type = IdType.AUTO)
    private Long id;

    private String name;
    private BigDecimal amount;
    private Integer status;

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;
}
```

**规则：**
- 必须用 `@Data` + `@TableName`
- 主键统一 `@TableId(type = IdType.AUTO) private Long id`
- 时间字段统一 `LocalDateTime`（不用 `Date` / `Timestamp`）
- `createTime` 用 `FieldFill.INSERT`，`updateTime` 用 `FieldFill.INSERT_UPDATE`
- 自定义列名用 `@TableField("column_name")`（如 `@TableField("is_read")`）
- 字段类型：`Long` 做 ID，`BigDecimal` 做金额，`Integer` 做状态/类型，`String` 做名称/描述

## 六、DTO / VO 规范

### DTO（请求参数）

```java
@Data
public class XxxDTO {
    @NotBlank(message = "名称不能为空")
    private String name;

    @NotNull(message = "金额不能为空")
    private BigDecimal amount;

    @NotBlank
    @Pattern(regexp = RegexUtils.PHONE_REGEX)
    private String phone;
}
```

- 使用 `jakarta.validation.constraints`（`@NotBlank` / `@NotNull` / `@Pattern`）
- 手机号正则引用 `RegexUtils.PHONE_REGEX` 常量，不写死
- 字段类型与 Entity 保持一致

### VO（响应数据）

```java
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class XxxVO {
    private Long id;
    private String name;
    private BigDecimal balance;
}
```

- VO 不加验证注解
- 字段类型与 Entity 保持一致（`Long` 不写成 `Integer`）

## 七、异常处理规范

```java
// 业务异常 — 在 Service 中抛出
throw new BusinessException(ResultConstant.BAD_REQUEST, "错误描述");
throw new BusinessException(ResultConstant.UNAUTHORIZED, "未授权");
throw new BusinessException(ResultConstant.SYSTEM_ERROR, "系统错误");

// 不要这样写
return Result.fail(code, msg);          // ❌
throw new RuntimeException(msg);        // ❌
```

**`GlobalExceptionHandler` 处理链：**

| 异常类型 | 日志级别 | HTTP 状态码 | 说明 |
|---------|---------|------------|------|
| `BusinessException` | `log.warn` | `e.getCode()` | 业务逻辑失败 |
| `MethodArgumentNotValidException` | `log.warn` | 400 | 参数校验失败 |
| `Exception` | `log.error` | 500 | 系统未预期异常 |

**日志级别：**
- 系统异常（DB/IO/网络）：`log.error`
- 业务异常（参数错误、余额不足）：`log.warn`
- 正常流程：`log.info`

## 八、缓存规范

### 三级缓存模式（Caffeine → Redis → DB）

```java
// 1. L1: 本地 Caffeine 缓存
List<VO> local = localCache.getIfPresent(key);
if (local != null) return local;

// 2. L2: Redis 缓存
CacheResult<List<VO>> redisResult = getFromRedis(key);
if (redisResult.isHit()) {
    localCache.put(key, redisResult.getData());
    return redisResult.getData();
}

// 3. L3: DB 查询
List<VO> dbData = loadFromDb();

// 4. 回填 L1 + L2
localCache.put(key, dbData);
if (!redisResult.isError()) {
    saveToRedis(key, dbData);
}
return dbData;
```

### Caffeine 缓存 Bean

全部定义在 `common/config/CaffeineCacheConfig.java`，通过 `@Resource(name = "...")` 注入：
```java
@Resource(name = "parkingLotListCache")
private Cache<String, List<ParkingLotVO>> parkingLotListCache;
```

### 管理端 Redis 降级

管理端 Redis 仅作为缓存使用，Redis 异常时可以降级查 DB（`log.warn` 不抛异常）。

### 客户端关键路径 Redis

- 登录/注册中 Redis 记录登录次数、封禁时间、验证码 —— 不可降级（降级将导致不安全）
- Redis 异常时直接抛 `BusinessException(500)`

## 九、Redis 规范

### CacheResult 三态处理

```java
CacheResult<String> result = redisService.get(key);
if (result.isError()) {
    log.warn("降级描述", e);         // 非关键路径：降级
    // 或
    log.error("关键错误", e);        // 关键路径
    throw new BusinessException(500, "系统繁忙");
} else if (result.isHit()) {
    use(result.getData());
} else {
    loadFromDb();                   // MISS：查 DB
}
```

### 默认使用 RedisService

优先用 `RedisService`（方法返回 `CacheResult<T>`），验证码模块例外可用 `StringRedisTemplate`。

直接用 `StringRedisTemplate` 时必须捕获并处理异常。

## 十一、MQ 规范

### Event 定义

```java
@Data
@AllArgsConstructor
public class XxxEvent implements Serializable {
    private Long id;
    private String data;
}
```

### Publisher

```java
@RabbitTemplate
private final RabbitTemplate rabbitTemplate;

public void sendXxx(XxxEvent event) {
    rabbitTemplate.convertAndSend(MQConstant.EXCHANGE, MQConstant.ROUTING_KEY, event);
}
```

### Consumer

```java
@RabbitListener(queues = MQConstant.XXX_QUEUE)
public void handle(XxxEvent event) {
    // 处理逻辑
}
```

### 死信延迟队列模式

1. **Source queue**（TTL 队列）：`x-dead-letter-exchange` + `x-message-ttl` 配置
2. **Proc queue**（处理队列）：消费者监听此队列
3. **Alert queue**（告警队列）：重试耗尽后转发到此队列，记录错误日志

### 重试模式

```java
if (retryCount < MQConstant.DEFAULT_RETRY_MAX) {
    // 重新入队延迟队列（retryCount + 1）
} else {
    // 转发到 alert 交换机
    rabbitTemplate.convertAndSend(MQConstant.RETRY_EXCHANGE, MQConstant.ALERT_ROUTING_KEY, event);
}
```

## 十二、WebSocket 规范

- 端点路径：`/ws/*/messages`
- 连接参数：`token`（从 URI 查询参数传递）
- 断线重连：URL 中带 `lastTimestamp` / `lastId` 参数，自动同步未读消息
- 推送链路：`Service` → `NotificationPublisher` → Redis Pub/Sub → `NotificationSubscriber` → `NotificationWebSocketHandler.pushToUser()`
- 用户多会话：每个 userId 对应 `CopyOnWriteArraySet<WebSocketSession>`（支持多设备/多标签）

## 十三、拦截器规范

### 三层拦截器链

| 顺序 | 拦截器 | 路径 | 行为 |
|------|--------|------|------|
| 0 | `RefreshTokenInterceptor` | `/**` | 有 token 则刷新 Redis session TTL，不阻塞请求 |
| 1 | `JwtTokenUserInterceptor` | `/orders/**`, `/wallet/**`, 等 | 验证用户 JWT + Redis session |
| 2 | `JwtTokenAdminInterceptor` | `/admin/**` | 验证管理员 JWT + Redis session |

### ThreadLocal

- `UserHolder` / `AdminHolder`：`ThreadLocal<Long>`，拦截器 `preHandle` 设置、`afterCompletion` 清理
- 接口中通过 `UserHolder.get()` 获取当前用户 ID

## 十四、定时任务规范

- 在 `common/config/SchedulingConfig` 中启用 `@EnableScheduling`
- 任务类声明为 `@Component @RequiredArgsConstructor @Slf4j`
- 异常用 try-catch 包裹（定时任务不能向外抛异常）

当前任务：

| 任务 | 频率 | 用途 |
|------|------|------|
| `CouponCleanupTask` | 每天 3:00 | 清理过期优惠券 Redis key |
| `DashboardPreAggregationTask` | 每 5 分钟 | 预计算今日订单数/收入 |
| `CacheConsistencyTask` | 每 10 分钟 | 重建可用车位、校验全量列表 |
